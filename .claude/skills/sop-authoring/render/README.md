# The Atlas SOP render engine

This is the **house engine** that gives every SOP the firm's visual seal: the
[Atlas Design System](../../../../brand/design-system/DESIGN.md) (petrol teal +
bronze on ivory; Source Serif 4 / IBM Plex Sans / IBM Plex Mono), built with the
[`impeccable`](../../impeccable/) skill. Producing this render is a **standard
part of authoring a SOP**, not an optional extra — see the parent
[`SKILL.md`](../SKILL.md).

**Why it exists:** the look must be *identical and durable* across sessions and
authors. Chat has no memory; the repo does. Because the stylesheet and fonts are
committed, shared assets, the only per-SOP input is the content — so whoever
asks for a SOP (Julia, Lilian, a future teammate) gets the same branded result,
deterministically.

## What's here

```
render/
├── README.md            ← you are here (authoring guide + component reference)
├── atlas.css            ← the stylesheet: tokens, components, light+dark, print. SHARED — do not fork per SOP.
├── shell.html           ← the generic scaffold: {{TITLE}} · {{STYLE}} · {{BODY}} + the theme/print/reveal script
├── build.mjs            ← deterministic assembler (body + shell + atlas.css + brand fonts → standalone + Artifact fragment)
└── examples/
    └── btr-body.html    ← the reference SOP body (the Business Tax Receipt render), authored with the classes below
```

Brand fonts live once, at
[`brand/design-system/fonts-embedded.css`](../../../../brand/design-system/fonts-embedded.css)
(Latin-subset woff2 as data URIs), regenerated by
[`brand/design-system/fetch-fonts.mjs`](../../../../brand/design-system/fetch-fonts.mjs).
`build.mjs` inlines them so every render is **self-contained** — it works
offline, in Google Drive, when printed, and inside a CSP-restricted Artifact
with zero external requests.

## How to render a SOP

1. **Author the body fragment** — an `.html` file with the visible markup only
   (toolbar → TOC → masthead → `<main>` → footer), using the classes in the
   reference below. Keep [`examples/btr-body.html`](./examples/btr-body.html)
   open as the pattern; copy its structure and swap the content. The Markdown
   SOP in `projects/sops/` stays the **source of truth** — the body fragment is
   a hand-laid *view* of it (the flowchart, the "what you pay" box, and the email
   cards are semantic, so a person maps them; they don't fall out of Markdown
   automatically).
2. **Build:**
   ```
   node .claude/skills/sop-authoring/render/build.mjs \
     --body    <path/to/body.html> \
     --title   "SOP · <Name> | JK Accounting Group" \
     --out     projects/sops/<sop-name>.html \
     --fragment <scratch>/<sop-name>.artifact.html   # optional
   ```
3. **Review copy:** publish the `--fragment` file with the **Artifact** tool
   (favicon `🧾`) so the reviewer gets a shareable light/dark link. Keep the
   **same Artifact file path** across review rounds so the URL is stable.
4. **Commit** the standalone `projects/sops/<sop-name>.html` next to its `.md`
   and note it in [`projects/sops/README.md`](../../../../projects/sops/README.md).

> **PDF is on request only.** The render has a Print / PDF button and a print
> stylesheet, so anyone can make a PDF in one click. Don't generate or send a PDF
> unless asked — the Artifact link is how the render is reviewed in chat.

When the `.md` changes, re-run `build.mjs` to refresh the view (edit the body
fragment first). When `atlas.css` improves, every SOP re-rendered from it picks
up the improvement — that's the point of a shared stylesheet.

## Component reference (the Atlas classes)

Compose only from these — they map straight to the SOP house structure in
[`SKILL.md`](../SKILL.md). Never invent colors or fonts; use the tokens.

| Structure (SKILL.md) | Markup |
|---|---|
| Toolbar (logo · print · theme) | `<header class="bar">` — generic; keep as-is |
| On-page index (desktop rail) | `<nav class="toc">` with `<a href="#id"><span class="sn">§N</span> …</a>` |
| Header block / cover | `<section class="mast">` → `.kick`, `<h1>` + `.loc`, `.lede`, `.meta` with `.chipm` (add `.live` for the Active status dot) |
| Section (§N) | `<section class="doc" id="sN">` → `<div class="shead"><span class="schip">§N</span><h2>…</h2></div>` + `.slede`. `.shead.big` for the flowchart header |
| Sub-header bar (3A/3B, Step A/B/Yes/No) | `<div class="subbar"><span class="tag">3A</span><h3>…</h3><span class="rt">…</span></div>` then children in `<div class="indent">` (left-rule indent) |
| Screen-by-screen unit | `<div class="screen"><div class="lab">Screen 1 <span class="t">Title</span></div>…</div>` |
| Numbered list (intake, questionnaire) | `<ol class="qlist">` (teal numbered chips) |
| Checklist / bulleted set | `<ul class="checks">` (bronze checks) · `<ul class="dots">` (teal dots) · `<ul class="sublist">` (sub-items) |
| Callouts (info/warn/error/ok/note) | `<div class="callout info\|warn\|err\|ok\|note">` → `.ic` (inline SVG) + `.cx` (`.cl` label + `<p>`; `.quote` for quoted text). Semantic tokens — separate from the bronze accent |
| "What you pay" money box | `<div class="paybox">` → `.ph` (`.coin` $ badge), `<ol class="pay-items">` (use `.amt` on figures), `.bottomline` |
| Process flowchart | `<figure class="flow">` → `.fnode` (`.gate`/`.pay`/`.wait`/`.done` variants, `.pill` tag, `.ckmark`), `.fconn` connectors, `.fphase`/`.fphase-h` phase groups, `.fdecide`/`.fbranch`/`.branch.fix`/`.branch.good` for decisions |
| Uploads / have-ready panel | `<div class="panel">` → `.ph`, `<h4>`, `.grp` groups |
| Email map (§7) | `.emailgrp` → `.eh` (`.from` sender pill), `.emailcard` (`.top` → `.step` + `.etag ack\|pay\|approve\|action`, `.subj`, `<p>`) |
| Contacts & links | `<div class="tablewrap"><table class="links">` (`.for` = mono "for" label, `.ph` = phone) |
| Blank intake appendix | `<div class="intake"><pre>…</pre></div>` with `.grp` on group headers |
| Footer | `<footer class="foot">` — carries the "the .md is the source of truth" note |

Hierarchy must read from ten feet away: `.schip` §-chips and serif `h2` for
sections; `.subbar` bars sit **above** their children; `.screen .lab` labels are
small mono kickers subordinate to their `.subbar` parent. Light + dark are both
handled by the tokens in `atlas.css`; the print stylesheet forces a clean light
copy — you don't style themes per SOP.
